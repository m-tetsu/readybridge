# ReadyBridge — AI相談（RAG）機能 設計メモ

> このファイルは、以前チャット上で相談したが記録に残っていなかった「AI相談（RAG）」機能の構想を、散逸しないよう書き起こしたもの。

## 1. やりたいこと（要件）

災害後は政府・自治体の補助金や支援制度が**随時追加・変更される**。被災した中小企業の経営者が、

- 「自社の状況（業種・規模・被害内容・地域）で、**どんな支援が受けられるか**」
- 「災害時に、**まず何をすべきか／どう申請するか**」

を、**最新の支援情報に基づき・出典付きで**相談できる仕組みをつくる。

ポイント：
- 中小企業庁などの公的な支援ページを**定期的に巡回**し、差分を取り込む（情報の鮮度が命）。
- 取り込んだ情報を **RAG（検索拡張生成）** の知識源にする。
- 回答は **安価なモデル（Claude Haiku）** を API 経由で呼び、**月数百〜数千円**のコストに収める。
- 災害支援は災害ごとに内容が変わる前提を**留保**として明示する（誤案内の防止）。

## 2. 全体アーキテクチャ

3つのコンポーネントに分かれる。CLAUDE.md のフェーズ分けと対応する。

```
┌─ ① 収集パイプライン（第2フェーズ）───────────────────────────┐
│  GitHub Actions (cron, 日次/週次)                            │
│   1. 公的ソースを取得（中小企業庁 等）                        │
│   2. 前回スナップショットと差分検知                          │
│   3. 本文を抽出・正規化 → Markdown 化                        │
│   4. チャンク分割 → 埋め込み生成 → ベクトルストアへ upsert    │
│   5. 重要な変更は PR を起票（人がレビュー）                  │
└──────────────────────────────────────────────────────────────┘
                          │ 知識ベース（ベクトル＋原文）
                          ▼
┌─ ② RAG 検索＋生成（第3フェーズ）─────────────────────────────┐
│  Cloudflare Worker（/api/chat）                              │
│   1. ユーザー質問 ＋ 状況（業種/規模/被害/地域）を受け取る   │
│   2. 質問を埋め込み → ベクトル検索で関連チャンク取得          │
│   3. 関連チャンク＋質問を Claude Haiku に渡す                 │
│   4. 出典リンク・最終更新日付きで回答を生成                  │
└──────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─ ③ フロントエンド（既存 chat.astro を実装に置き換え）─────────┐
│  src/pages/chat.astro（現在「準備中」プレースホルダ）        │
│   - 状況入力フォーム ＋ チャットUI                           │
│   - 回答に出典カード（タイトル・URL・更新日）を併記          │
└──────────────────────────────────────────────────────────────┘
```

## 3. 技術選定とトレードオフ

ホスティングが **Cloudflare Workers Static Assets** である前提で選ぶ。

| 役割 | 第一候補 | 理由 / トレードオフ |
|---|---|---|
| ベクトルストア | **Cloudflare Vectorize** | Workers とネイティブ統合・追加インフラ不要・無料枠あり。Cloudflare に寄せると運用が一番楽。代替：Turso/libSQL＋sqlite-vss、Supabase pgvector（別サービスが増える） |
| 埋め込み生成 | **Workers AI**（`@cf/baai/bge-m3` など多言語対応）| Worker 内で完結・日本語対応・Cloudflare 内で安価。代替：OpenAI/Cohere の埋め込みAPI（外部依存・別課金が増える） |
| 生成（LLM） | **Claude Haiku 4.5**（`claude-haiku-4-5`）| 安価（$1 / 100万入力トークン、$5 / 100万出力トークン）・200Kコンテキスト・日本語良好。出典に忠実な要約・抽出に十分。難しい相談だけ上位モデルにエスカレーションする設計も可 |
| 収集スケジューラ | **GitHub Actions cron** | 既存ワークフロー（push→Cloudflare自動ビルド）と親和。差分時に PR 起票でき、人のレビューを挟める |
| 本文抽出 | Readability 系 ＋ HTML→Markdown | 公的ページはHTMLが雑なこともあるので正規化が要る |

> モデルIDは正確を期す：Claude Haiku 4.5 ＝ `claude-haiku-4-5`（200Kコンテキスト、入力 $1 / 出力 $5 per 1M tokens）。最新・最上位は Opus 4.8（`claude-opus-4-8`）。難問のみ Opus にエスカレーションする2段構成も検討余地あり。

### Anthropic API 利用上の注意（Haiku 4.5）
- Haiku 4.5 は **`effort` パラメータ非対応**（Opus/Sonnet 4.6+ のみ）。指定するとエラー。
- 拡張思考が必要なら旧来の `thinking: {type:"enabled", budget_tokens:N}`（`budget_tokens < max_tokens`、最小1024）。ただしRAG要約用途では基本不要。
- `max_tokens` は非ストリーミングで ~16000 目安。
- レート制限プールは Haiku 3.x とは別。

## 4. データソース（巡回対象の候補）

`src/pages/sources.astro` の参照ソース一覧と整合させる。初期候補：

- 中小企業庁（被災中小企業支援、災害関連の補助金・給付・融資）

- 日本政策金融公庫（災害貸付）
- 経済産業省（災害時の支援パッケージ）
- 厚生労働省（雇用調整助成金 等：休業手当まわり）
- 各自治体（後フェーズ。災害ごとに対象自治体が変わるため別管理）

> 留意：robots.txt とアクセス頻度を守る。日次1回程度の低頻度で十分。各ソースの利用規約を確認。

## 5. 知識ベースのスキーマ（チャンク単位）

各チャンクに**出典メタデータを必ず持たせる**（回答の出典明示と鮮度表示のため）。

```jsonc
{
  "id": "chusho-saigai-hojokin-0003",
  "text": "（本文チャンク。500〜800字程度）",
  "embedding": [/* ベクトル */],
  "metadata": {
    "source_name": "中小企業庁",
    "title": "○○災害における中小企業者向け支援",
    "url": "https://www.chusho.meti.go.jp/...",
    "fetched_at": "2026-06-19",
    "updated_at": "2026-06-10",     // ソース側の最終更新（取れれば）
    "pillar": "recover",            // prepare / recover
    "category": "finance",          // categories.ts と対応
    "disaster_scope": "general"     // 特定災害名 or general
  }
}
```

## 6. 回答生成のプロンプト方針

- 渡した**関連チャンクのみ**を根拠に回答する（ハルシネーション抑制）。
- **出典（URL・最終更新日）を必ず併記**。根拠が無いことは「最新の公式情報を確認してください」と返す。
- 災害支援は**災害ごとに内容が変わる前提**を留保として添える。
- 専門用語をかみくだき、**次の一手（窓口・申請先）を具体的に**書く（サイト全体の方針と一致）。
- ユーザーの状況（業種・規模・被害・地域）で検索クエリと回答を絞り込む。

## 7. 概算コスト

- **埋め込み・ベクトル検索**：Workers AI / Vectorize の無料枠〜従量。低トラフィックなら月 ¥0〜数百円。
- **生成（Haiku 4.5）**：1相談あたり 入力 ~4–8K＋出力 ~1K トークン想定 ≒ $0.01 前後。
  - 月1,000相談でも **$10〜15（約¥1,500〜2,500）** 程度。
- **収集（GitHub Actions）**：パブリックリポジトリなら基本無料枠内。
- → CLAUDE.md の「公開後も月数百〜数千円」の見立てと整合。

## 8. 実装ステップ（提案）

1. **第2フェーズ：収集パイプラインの最小版**
   - 1〜2ソースだけ対象に、cron で取得→Markdown化→`src/content/` 配下に保存→差分時PR起票。
   - この時点では RAG なしでも「最新情報の差分検知」だけで価値が出る。
2. **ベクトル基盤の用意**
   - Vectorize インデックス作成、Workers AI 埋め込み、収集物を upsert するスクリプト。
3. **第3フェーズ：`/api/chat` Worker**
   - 質問→埋め込み→検索→Haiku 生成→出典付き回答。
4. **フロント実装**
   - `chat.astro` を状況入力フォーム＋チャットUIに置き換え、出典カードを表示。
5. **評価・調整**
   - 既知の良質な回答例で精度確認。プロンプトと検索件数（top-k）をチューニング。

## 9. 決定事項 / 次に決めること

### 決定済み
- **[確定] ベクトルストア = Cloudflare Vectorize**（2026-06-19）
  - 理由：現方針（Cloudflareに寄せる・運用負荷ほぼ0・低コスト）と完全一致。Worker から `env.VECTORIZE.query()` で直結でき、別サービスのAPIキー管理・ネットワーク設定が不要。この規模（数百〜数千チャンク）なら無料枠内で実質月¥0。埋め込み（Workers AI）と同一基盤で一気通貫。
  - ロックイン対策：原文Markdownを残せばベクトルは再生成可能なので、ロックインの実害は小さい。
  - 補足：数百チャンク程度の初期はベクトルDB不要の「ブルートフォース（KV/R2に埋め込み保存＋Worker内でコサイン類似度）」でも動くが、巡回ソース追加ですぐ限界が来るため、二度手間を避け最初からVectorizeを採用する。

- **[確定] 埋め込み = Workers AI `@cf/baai/bge-m3`**（2026-06-19）
  - 理由：多言語特化で日本語に強く、この用途の精度要件を満たす。Vectorize/Worker と同一基盤でHTTP往復・APIキー追加が不要（論点⑤の負担も減る）。この規模では外部高精度モデルとの差が体感に出にくい。
  - 留保：**検索品質は必ず実データで評価**する（良質な想定問答セットで上位k件の妥当性を確認）。不十分なら (a) チャンク分割の見直し → (b) Cohere `embed-multilingual-v3` 等への移行、の順で対処。原文Markdownから再生成できるため移行リスクは小さい。
  - 制約：ベクトルは保存時と検索時で**同一モデル**を使う（途中変更時は全チャンク再生成）。

- **[確定] 巡回対象ソースの初期リスト**（2026-06-19・暫定。後日精査して追加/変更予定）
  - 方針：3層に分け、**A層（恒常的な支援の土台）中心**で開始。B層（災害特設）は枠だけ用意、C層（自治体）は当面スコープ外。
  - **初期リスト（A層）：**
    1. 中小企業庁（経営サポート＞災害対策） — recover全般・窓口
    2. 日本政策金融公庫（災害貸付） — finance
    3. 中小機構（共済・経営相談） — recover
    4. 厚生労働省（雇用調整助成金） — wage
    5. 国税庁（災害時の納税の猶予・減免） — tax
    6. 経済産業省（被災者支援パッケージの入口） — recover全般・導線
  - **B層（災害特設）**：ソース定義に `disaster_scope` を持たせ、災害発生時に該当URLを追加するだけで巡回対象に入る設計（§5メタデータと整合）。
  - **C層（自治体）**：数が多く構造もバラバラなため後フェーズ。
  - **robots / 規約の運用ルール：**
    - 各サイトの `/robots.txt` を尊重（Disallow対象は巡回しない）。
    - 低頻度（日次1回程度）・`User-Agent` 明示・リクエスト間隔を空ける。
    - 利用規約・再利用条件を**ソースごとに個別確認**（多くは政府標準利用規約相当で出典明示すれば可、ただし要確認）。
    - **全文転載しない**。知識ベースは要点チャンク＋出典を持ち、回答画面では原文へリンク誘導。
    - 取得失敗・構造変化を検知したら取り込まず**スキップ＋通知**（誤情報の自動公開を防ぐ）。
    - 差分があった時だけPR起票し、人のレビューを挟む。

- **[確定] 免責表示 = 3点セットで担保**（2026-06-19）
  - **① 常時表示（UI固定）**：チャット入力欄付近に常設のスティッキー注記。例「本回答はAIによる参考情報です。支援制度は災害・時期により変わります。申請前に必ず出典の公式ページと窓口でご確認ください。」落ち着いた紙色＋小さめの注記で（警告バナー化しない）。
  - **② 回答ごとの根拠（実質の核）**：各回答末尾に**出典カード（ソース名・タイトル・URL・最終更新日/取得日）を必須**。これを満たせない回答は出さない。根拠が無い質問は断定せず「手元の最新情報では確認できませんでした。○○窓口へお問い合わせください」と返す。
  - **③ 鮮度・確信度の明示**：回答に「情報取得日」を表示。災害特設（B層・`disaster_scope`）由来には「特定災害向けの制度です。対象・期限をご確認ください」の但し書きを自動付与。
  - トーン：「免責」ではなく「確認のお願い」。怖がらせず必ず目に入る配置。
  - 実装分担：①は `chat.astro` のUI、②③は `/api/chat` の回答整形。

- **[確定] Anthropic APIキー管理 = Workers Secrets 一本**（2026-06-19）
  - 論点①②でCloudflareに寄せた結果、鍵管理対象は実質**Anthropicキー1つだけ**（埋め込みはWorkers AIで鍵不要）。
  - **本番**：`ANTHROPIC_API_KEY` を **Workers Secrets（Encrypted）**で保存（`wrangler secret put` or ダッシュボードのEncrypted変数）。Worker内は `env.ANTHROPIC_API_KEY` で参照。`wrangler.toml` の `vars`（平文）には**絶対に書かない**。
  - **ローカル**：`.dev.vars` に記載し**必ず `.gitignore` に追加**（RAG着手時に最初に対応）。
  - **権限/ローテーション**：Anthropicコンソールで**用途別の専用キー**を発行。漏洩時は Secrets 再登録だけでローテーション完結（コード変更不要）。
  - **コスト保険**：Anthropic側で使用上限を設定＋Worker側で `max_tokens` 上限と簡易レート制限（KVで回数管理）。
  - **セキュリティ**：キーはサーバー（Worker）側のみで使用、フロントに渡さない。ログにキー・生レスポンスを出さない。
